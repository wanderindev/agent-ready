# Make the good path the default

If there is one idea the rest of this project keeps circling back to, it is this: where a good practice depends on a person choosing it under pressure, the practice loses. Not because people are careless, but because the moments the practice matters most, the deadline, the tired end of a long session, the twentieth nearly-identical diff, are exactly the moments discretion gets spent elsewhere. So the durable move is to remove the choice. Make the good path the one that happens by default, and the careful thing the thing you have to go out of your way to skip.

## Gates instead of guidelines

Concretely, that means preferring a structural gate to a written rule whenever a skill is running at the moment the rule applies. A guideline is a sentence in a document, and documents are not reread under pressure. A gate is a refusal, and a refusal holds regardless of how tired the operator is.

The methodology is full of them, and I built each one at the point where I noticed a rule was about to be ignored. The audit prompt will not assemble until its per-area specifics are filled in, because "please ground your examples in this area" was the step most likely to be skipped. A brief cannot be dispatched until it passes a well-formedness check with no unfilled placeholders and an explicit statement that it touches no production path, because a half-written brief is the one that produces a bad autonomous change. A pull request cannot be marked ready until an [independent reviewer](03-you-cant-review-your-own-work.md) passes it. None of these are advice. They are places the pipeline stops and refuses to continue until the careful thing has actually happened.

The team version sharpened this for me. On the pilot, one developer chose to write tests and keep docs current, and the rest mostly did not, because it was a choice and choices under deadline go the cheap way. The answer was not to exhort people to write tests. It was a gate that refuses to open a pull request until the documentation check and the tests on the meaningful paths are green, and a session hook that will not let a work session end while lint or tests are failing. Once those exist, good tests and current docs stop being a personal virtue and become the default state, the thing that happens unless you deliberately break it. That is a much more reliable way to raise a floor than asking people to be diligent.

There is an honest limit here, and I want to name it because it is a real part of the design. Not every discipline can be gated. Two of the methodology's most important practices, preserving the exact prompt from every session and keeping a running log of cross-session decisions, happen in the gaps between sessions, where no skill is running and so no gate can fire. Those stay written conventions, and they decay the way conventions do, and I catch the decay after the fact rather than preventing it. Knowing which disciplines you can gate and which you can only ask for is itself part of the method. The test I use is whether a skill is in the loop at the exact moment the discipline needs to apply. If it is, build the gate. If it is not, be honest that you are relying on a habit.

## One command, run it again to fix it

The other half of making the good path the default is making the good setup effortless to get into and painless to repair. That is what the onboarding skill does: one command that checks your prerequisites, installs the [safety floor](02-a-safety-floor-you-commit-not-remember.md), makes sure the target repo is bootstrapped and documented, and then health-checks the whole result. Every step is built to be run again. Re-running it on a half-configured machine is not a special repair mode, it is the ordinary way you fix a setup that drifted.

The design principle underneath it came straight from the team work: automate everything that can be automated, and for the few parts that genuinely need a human, a token to create, an admin action to take, give the exact step and then verify it, rather than leaving the person to figure out whether it worked. The skill is deliberately thin. It does not reimplement the bootstrap or the docs install, it sequences them and adds the machine-level pieces they were missing. Its job is not to do new work. Its job is to make sure none of the existing work gets skipped, and to make getting set up a two-minute command instead of a checklist you follow imperfectly.

## Why a team cannot just copy your files

Standing behind both the gates and the one-command install is a distinction between a solo setup and a shared one that took me a while to see clearly. A single developer can configure their environment by hand and copy a good file from an old project into a new one. A team cannot, and not because they are less capable. Copying is a snapshot. The moment you improve the shared config, every copy is stale, and there is no path for the fix to reach the machines that need it. A group needs a channel, not a snapshot, which is why the whole thing is distributed as installable plugins with an update path rather than as files to clone. Fix the guardrail policy once, and everyone picks it up with an update, instead of the fix living on one machine and dying there.

## Adoption is not the license you bought

The last thing this page is really about is a mistake that is easy to make and expensive to keep making. The simplest thing an organization can do about AI is buy everyone a subscription, and it is the wrong place to stop, because a pile of individual licenses is not a shared way of working. It is a pile of individual habits, most of them undocumented, none of them improving each other.

The leverage is not in the access. It is in making the good way of working the shared default, and in making context collective instead of private. When one person spends an afternoon understanding a gnarly part of the system, that understanding should become something the whole team inherits, a durable asset in a place everyone can reach, not an answer that lived in one private chat and evaporated when the tab closed. Buying the license is the easy 10 percent. Making the careful path the default one, and making hard-won context something the group keeps, is the other 90, and it is the part that actually changes how the work goes.
